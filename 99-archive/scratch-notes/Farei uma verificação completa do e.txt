##  Auditoria das Regras e Mecanismos do NeoCortex

### 1. STEP 0 (Regression Check)

| Aspecto | Detalhe |
| :--- | :--- |
| **O que ** | Verificao obrigatria antes de qualquer ao, consultando o `Regression Buffer` para evitar repetir erros. |
| **Onde est implementado** | Ferramenta MCP `neocortex_regression` (ao `check`). <br> Arquivo: `neocortex/mcp/tools/regression.py` |
| **Como deve ser invocado** | O agente (T0) deve chamar `neocortex_regression.check(task_description)` antes de executar qualquer ao de escrita ou modificao. |
| **Validao** | Verificar se a ferramenta est registrada no `server.py` e se o mtodo `check` consulta o `RegressionService`. |

```python
    if action == "check":
        # Consulta o RegressionService e retorna similaridades
        ...
```

### 2. Modo Mentor (STEP 0 Forado para Agentes Locais)

| Aspecto | Detalhe |
| :--- | :--- |

| **Onde deve estar implementado** | Funo `mentor_step_0()` dentro de `neocortex/mcp/sub_server.py`. <br> **Status:**  Planejada (AGENT-203), **no totalmente implementada.** |

### 3. Atomic Locks

| Aspecto | Detalhe |
| :--- | :--- |
| **O que ** | Lista de arquivos/diretrios que **nenhum** agente pode modificar sem autorizao explcita. |
| **Onde est implementado** | Definido no Crtex (`00-cortex.mdc` ou `config.yaml`). Aplicado pelo `SecurityService`. <br> Ferramenta MCP: `neocortex_security.validate_access`. |
| **Como deve ser usado** | Antes de qualquer modificao em arquivo, o sistema valida se o caminho est na lista de `atomic_locks`. |
| **Validao** | Verificar se o `SecurityService` carrega a lista de locks do `ConfigProvider` e se a ferramenta `validate_access` retorna `False` para recursos bloqueados. |

---

### 4. Polticas de Visibilidade Hierrquica

| Aspecto | Detalhe |
| :--- | :--- |
| **O que ** | Regras que definem o que um agente pode ver: nveis inferiores, iguais (com permisso), mas nunca superiores. |
| **Onde est implementado** | Planejado para a ferramenta `neocortex_visibility` e `neocortex_hierarchy`. <br> **Status:**  No implementado (HIER-001 a HIER-005). |
| **Como deve ser implementado** | Cada Lobo/agente tem um `level` e `parent_id`. Funo `can_access(user_level, resource_level, user_id, owner_id)`. |
| **Validao** | Verificar se os arquivos de configurao de agente (`lobes/*/config.yaml`) contm campos `hierarchy.level` e `hierarchy.parent_id`. |

---

### 5. Polticas de Uso por Agente (Rate Limiting, Token Budget)

| Aspecto | Detalhe |
| :--- | :--- |
| **O que ** | Limites de tokens por tarefa, tempo mximo de execuo, oramento dirio. |
| **Onde est implementado** | Planejado para `neocortex_failsafe` e `ConfigProvider`. <br> **Status:**  No implementado (SEC-403). |
| **Como deve ser implementado** | No `sub_server.py`, antes de enviar ao LLM, verificar se a tarefa excede `max_tokens_per_task` ou se o `daily_budget` foi atingido. |
| **Validao** | Verificar se `config.yaml` do agente contm chaves `limits.max_tokens_per_task` e `limits.daily_budget`. |

---

### 6. Fallback Chain (Cadeia de Fallback para LLMs)

| Aspecto | Detalhe |
| :--- | :--- |
| **O que ** | Sequncia de backends LLM a serem tentados em caso de falha. |
| **Onde est implementado** | `LLMBackendFactory` e `ConfigProvider`. <br> **Status:**  Implementado (LLM-006). |
| **Como funciona** | Se o backend primrio falhar (timeout, erro), a factory tenta o prximo da lista `fallback_chain`. |
| **Validao** | Verificar se `config.yaml` contm a chave `fallback_chain` e se o `AgentExecutor` a utiliza. |

---

### 7. Roteamento Inteligente por Role/Complexidade

| Aspecto | Detalhe |
| :--- | :--- |
| **O que ** | Deciso automtica de qual modelo LLM usar baseado no papel do agente ou na complexidade da tarefa. |
| **Onde est implementado** | Planejado para o `T0 Autnomo` e `neocortex_brain`. <br> **Status:**  Parcial (backend_override no spawn, mas sem deciso automtica). |
| **Como deve ser implementado** | O T0 analisa a tarefa, consulta polticas no `ConfigProvider`, e decide o backend. |
| **Validao** | Verificar se existe uma ferramenta `neocortex_brain.think` que retorna uma deciso de roteamento. |

---

### 8. Heartbeat e PulseScheduler (Manuteno Autnoma)

| Aspecto | Detalhe |
| :--- | :--- |
| **O que ** | Sistema que mantm a sade do contexto: checkpoints peridicos, pruning, consolidao, backups. |
| **Onde est implementado** | `PulseScheduler` em `neocortex/core/pulse_scheduler.py`. <br> **Status:**  Implementado e integrado (OPT-010). |
| **Como funciona** | Tarefas agendadas: checkpoint (5min), pruning (15min), consolidao (30min), AKL (dirio), backup (dirio). |
| **Validao** | Verificar logs do servidor MCP para mensagens `[Pulse] ...`. |

---

### 9. Auditoria e Audit Trail

| Aspecto | Detalhe |
| :--- | :--- |
| **O que ** | Registro imutvel de todas as aes (quem fez o qu, quando, resultado). |
| **Onde est implementado** | `AuditTrail` dentro do `LedgerService`. <br> **Status:**  Parcial (servio existe, mas no  invocado em todas as aes). |
| **Como deve ser implementado** | Cada ferramenta MCP que modifica estado deve chamar `LedgerService.add_audit_entry()`. |
| **Validao** | Verificar se o `$LDG` contm uma seo `audit_trail` com entradas recentes. |

---

### 10. Consolidao Semntica e AKL

| Aspecto | Detalhe |
| :--- | :--- |
| **O que ** | Transformao de logs brutos em regras permanentes (Consolidao) e decaimento de conhecimento obsoleto (AKL). |
| **Onde est implementado** | `ConsolidationService` e `AKLService`. Ferramentas MCP: `neocortex_consolidation`, `neocortex_akl`. <br> **Status:**  Servios prontos, agendados pelo PulseScheduler. |
| **Como funciona** | O PulseScheduler chama `consolidation.summarize_session()` a cada 30min e `akl.assess_importance()` diariamente. |
| **Validao** | Verificar se os logs de consolidao aparecem no `PulseScheduler` e se o Crtex  atualizado com novas regras. |
